# myapp — AWS Infrastructure & CI/CD

A production-grade AWS infrastructure project deploying a containerised Flask API to EC2, backed by PostgreSQL RDS, managed by Terraform, with a full CI/CD pipeline using GitHub Actions and Docker Hub.

---

## Architecture Overview

```
Internet
   │
   ▼
┌──────────────────────────────────────┐
│  EC2 Staging  (t3.micro)             │  ← Auto-deployed on every push to main
│  Public subnet us-east-1a            │
│  Docker: myapp container on port 80  │
└──────────────────────────────────────┘

┌──────────────────────────────────────┐
│  EC2 Production  (t3.small)          │  ← Manual deploy via workflow_dispatch
│  Public subnet us-east-1b            │
│  Docker: myapp container on port 80  │
└──────────────────────────────────────┘

Both EC2 instances connect to:

┌──────────────────────────────────────┐
│  RDS PostgreSQL 15  (db.t3.micro)    │  ← Private subnets, encrypted at rest
│  Shared between staging + production │
└──────────────────────────────────────┘
```

### Component decisions

| Component | Choice | Why |
|-----------|--------|-----|
| **Compute** | EC2 (Docker) | Simple SSH-based deploy, no container orchestration overhead |
| **Registry** | Docker Hub | No AWS-specific setup; works with standard `docker push/pull` |
| **Database** | RDS PostgreSQL 15 | Managed backups, encryption at rest, no self-managed replication |
| **Secret storage** | AWS Secrets Manager | DATABASE_URL injected at deploy time via EC2 instance role; never in GitHub secrets |
| **SSH keys** | Terraform `tls_private_key` | Key pair generated by Terraform; PEM output stored as GitHub secret |

## Prerequisites

| Tool | Version |
|------|---------|
| AWS CLI | 2.x |
| Terraform | 1.5+ |
| Docker | 24+ |

You also need:
- An AWS account with permissions for VPC, EC2, RDS, Secrets Manager, IAM, CloudWatch.
- A Docker Hub account with a repository named `myapp`.
- A GitHub repository with Actions enabled.

## Deploy the Infrastructure

```bash
cd terraform

terraform init
terraform plan
terraform apply
```

After `apply`, note the outputs:

```bash
terraform output staging_public_ip      # → STAGING_EC2_IP
terraform output production_public_ip   # → PROD_EC2_IP
terraform output -raw ec2_private_key_pem  # → EC2_SSH_KEY
```

---

## Configure GitHub Secrets

In **Settings → Secrets and variables → Actions**, add:

| Secret | Value |
|--------|-------|
| `DOCKERHUB_USERNAME` | Your Docker Hub username |
| `DOCKERHUB_TOKEN` | Docker Hub personal access token |
| `STAGING_EC2_IP` | From `terraform output staging_public_ip` |
| `PROD_EC2_IP` | From `terraform output production_public_ip` |
| `EC2_SSH_KEY` | From `terraform output -raw ec2_private_key_pem` |

---

## Configure the `production` GitHub Environment

1. Go to **Settings → Environments → New environment**, name it `production`.
2. Enable **Required reviewers** and add at least one reviewer.
3. The production deploy workflow pauses at `environment: production` until a reviewer approves.

## CI/CD Flow

### Pipeline 1 + 2: `ci.yml` — Build & Deploy Staging

Triggered automatically on every push to `main`.

**Job 1 — Build & Push:**
1. Builds Docker image from `app/`.
2. Tags image as `sha-<commit-sha>`.
3. Runs Trivy vulnerability scan (reports HIGH/CRITICAL, does not block).
4. Pushes image to Docker Hub.

**Job 2 — Deploy to Staging** (runs after build succeeds):
1. SSHs into the staging EC2 instance.
2. Fetches `DATABASE_URL` from AWS Secrets Manager using the EC2 instance role.
3. Pulls the newly built image tag.
4. Stops the old container, starts the new one on port 80.
5. Smoke-tests `/health` — must return HTTP 200.

### Pipeline 3: `deploy-prod.yml` — Deploy to Production

Triggered manually via **Actions → Deploy to Production → Run workflow**.

- **Input:** image tag (optional — leave blank to auto-resolve latest from Docker Hub).
- Shows last 10 available Docker Hub tags before deploying.
- Pauses at the `production` environment gate for manual approval.
- Deploys to production EC2 with the same SSH + Secrets Manager pattern.
- Health check loop: 6 attempts × 10 seconds.

## Monitoring

All monitoring resources are provisioned by Terraform in `terraform/monitoring.tf`.

### CloudWatch Dashboard

One dashboard: `myapp-overview`

| Section | Metrics |
|---------|---------|
| **EC2 Application** | CPU utilisation (staging + prod), Network In/Out |
| **RDS Infrastructure** | CPU, Database connections, Free storage space, Read/Write latency |
| **Alarms** | Status widget for all 4 alarms |

### CloudWatch Alarms

All alarms notify via SNS (set `alarm_email` variable to subscribe):

| Alarm | Condition |
|-------|-----------|
| `myapp-staging-cpu-high` | Staging EC2 CPU > 80% for 10 min |
| `myapp-prod-cpu-high` | Production EC2 CPU > 80% for 10 min |
| `myapp-rds-cpu-high` | RDS CPU > 80% for 10 min |
| `myapp-rds-low-storage` | RDS free storage < 5 GiB |

## Security

| Decision | Rationale |
|----------|-----------|
| **No hardcoded secrets** | DB password generated by `random_password`; stored in Secrets Manager only |
| **EC2 instance role** | EC2 reads Secrets Manager via IAM role — no credentials in GitHub or shell history |
| **Private RDS** | RDS lives in private subnets; only EC2 security groups can reach port 5432 |
| **Non-root container** | Dockerfile drops to UID 1000 before running gunicorn |
| **Trivy scanning** | Every image scanned for HIGH/CRITICAL CVEs before push |

## Local Development

```bash
# Start PostgreSQL + app locally
docker compose up --build

# Health check
curl http://localhost:8080/health

# Create an item
curl -X POST http://localhost:8080/items \
  -H "Content-Type: application/json" \
  -d '{"name": "hello world"}'

# List items
curl http://localhost:8080/items

# Stop
docker compose down -v
```

The `docker-compose.yml` sets `DATABASE_URL` directly — Secrets Manager is not used locally.
